1 · System Overview

StockFlow is a multi-tenant inventory SaaS MVP. One Express + Prisma + SQLite API. One React + Vite SPA. JWT in an httpOnly cookie. Every read/write is scoped by organizationId derived from the session.

Browser (React SPA, port 5173 via Vite proxy) │ fetch /api/* with credentials:'include' (cookie carries JWT) ▼ Express API (port 4000) │ cookieParser → requireAuth → requireRole → route handler ▼ Prisma Client → SQLite file (prisma/dev.db)

In production, the same Express process serves the built SPA from client/dist, so it's a single deployable artifact (server/src/index.ts).

2 · Data Model — prisma/schema.prisma

ModelKey columnsNotes
Organizationname, defaultLowStockThresholdtenant root
Useremail @unique, passwordHash, organizationId, rolebelongs to one org
ProductorganizationId, name, sku, quantityOnHand, costPrice, sellingPrice, lowStockThreshold, lastUpdatedBy@@unique([organizationId, sku]), @@index([organizationId])

Architect's notes

  • Shared-DB, shared-schema multi-tenancy with a discriminator (organizationId). Cheap; you must enforce scope in every query.
  • SKU is unique per org, not globally — correct for multi-tenant.
  • Product.lowStockThreshold nullable; null means "use org default" — classic override-or-fallback.
  • onDelete: Cascade from Organization → User/Product is destructive; in real life prefer soft-delete.
  • lastUpdatedBy stores the email string, not a userId. Cheap audit; loses referential integrity if the user changes email.
  • Prisma generator is pinned to ../server/node_modules/.prisma/client so the workspace doesn't generate into the wrong tree (a real bug fixed in this codebase).

3 · Backend Walkthrough — server/src/

3.1 Bootstrap — index.ts

express() → cors({ origin: env.corsOrigin, credentials: true }) // needed for cookies → express.json({ limit: "1mb" }) → cookieParser() → /api/health, /api/auth, /api/products, /api/settings, /api/dashboard, /api/users → static client/dist + SPA fallback (regex excludes /api/) → centralized error handler (logs + 500)
  • CORS + credentials — must explicitly list origin; * is rejected by browsers when credentials: true.
  • SPA fallback uses /^(?!\/api\/).*/ so deep links return index.html, but /api/... still 404s if not matched.
  • Gaps for prod: no request-ID middleware, no structured logger, no rate limiter.

3.2 Environment — env.ts

Throws on missing JWT_SECRET at import time → fail-fast. corsOrigin is comma-separated. Defaults for port and nodeEnv. 12-factor friendly.

3.3 Database — db.ts

Single PrismaClient instance. One process = one connection pool. tsx-watch kills the process between reloads so no singleton trick needed.

3.4 Auth Core — auth.ts

Constants & types

  • COOKIE_NAME = "sf_session", MAX_AGE_MS = 7 days.
  • Role = "ADMIN" | "MANAGER" | "GUEST" + ROLES array.
  • SessionPayload = { userId, organizationId, email, role } — JWT body.

Functions

  • signSession(payload) — jwt.sign(payload, env.jwtSecret, { expiresIn: "7d" }). HS256, no kid, no rotation.
  • setSessionCookie(res, token) — httpOnly (XSS-resistant), secure in prod, sameSite: "lax" in dev / "none" in prod, 7-day maxAge.
  • clearSessionCookie(res) — pairs with logout.
  • readSession(req) — returns SessionPayload | null; catches jwt.verify errors.
  • requireAuth(req, res, next) — async. Reads session, 401 if missing. Back-compat: if JWT lacks role (signed before roles existed), fetch from DB. Real bug fixed in this codebase.
  • requireRole(...allowed) — higher-order middleware, same DB-fallback, 403 if role not allowed.

Why two layers? Composition. Read endpoints get just auth; write endpoints chain both. Cheaper to reason about than one mega-middleware.

3.5 Validation — validation.ts

Zod schemas at the boundary. Custom transformers turn "", null, and string numbers into proper number | null. productUpdateSchema = productCreateSchema.partial(). TypeScript trusts you; the network doesn't.

3.6 Auth Routes — auth.routes.ts

RouteAuthBehavior
POST /signuppubliccreates Org + User in a nested write, bcrypt(10), signs JWT, sets cookie, returns user.
POST /loginpublicfinds user by email, bcrypt.compare, issues JWT.
POST /logoutpublicclears cookie.
GET /merequireAuthreturns { user, organization }.
GET /sessionpublicalways 200; returns { user: null } if not logged in. Re-reads role from DB so role changes propagate without re-login.

Key design decision: /session returns 200-with-null rather than 401 so the SPA bootstrap doesn't have to treat "not logged in yet" as an error.

3.7 Products Routes — products.routes.ts

const canWrite  = requireRole("ADMIN", "MANAGER");
const canDelete = requireRole("ADMIN");
  • GET / — list, ?q= search across name+sku. Org-scoped.
  • POST / — canWrite. Validates body, checks SKU uniqueness, sets lastUpdatedBy.
  • GET /:id, PUT /:id, DELETE /:id — use findOwned() helper to enforce tenancy. Critical pattern: never trust id from path without org filter (IDOR).
  • POST /:id/adjust — quantityOnHand + delta. Rejects negative results. race Two concurrent +1s can lose an update. Should use Prisma's atomic { increment: delta }.

3.8 Settings Routes — settings.routes.ts

  • GET / — any auth user.
  • PUT / — ADMIN only; updates defaultLowStockThreshold.

3.9 Users Routes — users.routes.ts

  • GET / — any user can list the org's users (for UI).
  • POST / — ADMIN. Auto-generated temp password via crypto.randomBytes(4).toString("hex") + crypto.randomInt(10,100) (~36 bits). Returned once; hashed with bcrypt; never re-retrievable.
  • PATCH /:id — ADMIN. Org-scoped. Cannot demote the last ADMIN. race count-then-update without a transaction.
  • DELETE /:id — ADMIN. Cannot delete self. Cannot delete last ADMIN.

3.10 Dashboard — dashboard.routes.ts

One endpoint returns totals + low-stock list. Computed in app code (reduce, filter) rather than SQL aggregation — fine for MVP scale; at 100k products push to groupBy/count.

4 · Frontend Walkthrough — client/src/

4.1 Bootstrap — main.tsx

<StrictMode><BrowserRouter><AuthProvider><App/></AuthProvider></BrowserRouter></StrictMode>. Order matters because App calls useAuth.

4.2 Routing — App.tsx

  • <RequireAuth> — redirect to /login if no user.
  • <PublicOnly> — redirect away from /login//signup if authed.
  • Both wait on loading to avoid flash-of-wrong-route during initial /session fetch.

4.3 Auth Context — auth.tsx

AuthProvider keeps { user, organization, loading } and exposes login/signup/logout/refresh. useEffect on mount calls refresh() → /api/auth/session. Mutations call refresh() after success.

Why context, not Redux? Low cardinality, single source of truth, infrequent updates. Adding Redux/Zustand would be overkill.

4.4 API Client — api.ts

credentials: "include"        // browser attaches httpOnly cookie
Content-Type: application/json
throws HttpError(status, msg) on non-2xx
returns parsed JSON typed by <T>

The narrow waist of all server calls — auth headers, retries, logging, telemetry can all centralize here.

6 · Strengths to Highlight

  1. Boundary discipline — Zod on input, typed api<T>() on output, Prisma types in the middle.
  2. Tenant scoping centralized in helpers like findOwned() — easy to audit.
  3. httpOnly cookie + sameSite instead of localStorage JWT.
  4. Defense in depth for RBAC: UI hides, middleware blocks, "last admin" / "delete self" invariants protected on the server even if the UI lies.
  5. Single deploy artifact in prod (Express serves SPA) — cheap to host.
  6. /api/auth/session returns 200-with-null — small but important UX detail.

7 · Honest Weaknesses

  • security No CSRF token. SameSite=lax catches most attacks but not all.
  • security No rate limiting on /login / /signup → credential stuffing risk.
  • crypto bcrypt rounds = 10; raise to 12 in prod.
  • feature No password reset / email verification.
  • correctness Race conditions in stock adjust and "last admin" invariants. Fix with atomic Prisma ops + transactions.
  • infra SQLite is single-writer — swap to Postgres for concurrency.
  • audit No audit log beyond lastUpdatedBy.
  • perf Client-side search on Products; server already supports ?q=.
  • UX Inline role dropdown PATCHes on every change with no confirm.
  • security JWT can't be revoked before expiry — logout only clears the cookie client-side.
  • UX Temp password in HTTP body — fine over HTTPS but ideally a separate channel (email).

System & Architecture

1. Walk me through what happens when a user clicks "Sign in".
Form → auth.login() → POST /api/auth/login with credentials:'include' → loginSchema validates → bcrypt.compare → signSession → Set-Cookie sf_session (httpOnly) → SPA calls refresh() → GET /session → state updates → navigate('/dashboard').
2. Why JWT in a cookie vs Authorization: Bearer in localStorage?
httpOnly cookie isn't readable by JS → XSS can't steal the token. Cost: must handle CSRF (mitigated here with sameSite). Bearer/localStorage is simpler for SPAs but every XSS is game over.
3. How is multi-tenancy enforced?
Single column organizationId. Every query filters by req.session.organizationId. findOwned() helper enforces ownership on lookups. Risk: any forgotten filter is a tenancy leak. Hardening path: a Prisma extension ($extends.query) that injects organizationId from AsyncLocalStorage, or row-level security on Postgres.
4. Where could one tenant read another's data today?
Anywhere req.params.id is used without scoping. The code does scope everywhere, but a global Prisma middleware would be belt-and-braces.

Auth & RBAC

5. Walk me through the RBAC matrix.
ADMIN: all. MANAGER: create/update/adjust products; no delete; no settings; no user mgmt. GUEST: read-only. Enforced in requireRole on the server and mirrored in permissions.ts for UI gating.
6. A user is downgraded ADMIN → GUEST while logged in. What happens?
Their JWT still says ADMIN until it's reissued or /session re-reads role from DB. The server requireRole only re-reads from DB if the claim is missing. Real fix: always re-read role from DB on guard, or include a tokenVersion in JWT and bump it on role changes.
7. How would you revoke a JWT immediately?
Add User.tokenVersion: Int. Embed in the JWT. Compare on guard. Increment on role change / forced logout. Or move to opaque session IDs in a Redis store.
8. Why is requireAuth async now?
To support back-compat with tokens that pre-date the role field. It hits Prisma only when the claim is missing — zero overhead for fresh tokens.

Data & Integrity

9. Walk me through the +1 stock adjust path. What's wrong?
Reads quantityOnHand, computes next = current + delta, updates. Two concurrent calls can both read N and both write N+1 → lost update. Fix: prisma.product.update({ data: { quantityOnHand: { increment: delta } } }) (atomic at DB), or wrap in a serializable transaction.
10. Prevent qty going below zero atomically. How?
DB CHECK constraint (quantityOnHand >= 0) + atomic increment → map constraint violation to 400. Or conditional update WHERE quantityOnHand + delta >= 0 and check affected rows.
11. Why @@unique([organizationId, sku]) and not just @unique?
Tenants must be free to choose their own SKUs without colliding across orgs.
12. Why are costPrice / sellingPrice Float? Risk?
Float = IEEE 754 = rounding errors for money. Use Decimal (Postgres numeric) or integer cents.

API Design

13. Why does /session return 200 with null instead of 401?
It's the SPA's initial bootstrap call. 401 would force "not-logged-in-yet" to be handled as an error. A null-user 200 cleanly encodes absence-of-session in the response shape.
14. POST /products can 409. How to make it robust?
Current code pre-checks then creates — TOCTOU. Better: rely on the DB unique constraint and catch Prisma's P2002, returning 409. Eliminates the race.
15. CORS: why credentials: true and an explicit origin?
Browsers block * with credentials. Must enumerate allowed origins and set the credentials flag. Configured via CORS_ORIGIN env var.

Frontend

16. Why context for auth instead of Redux?
Low write frequency, no complex selectors, no time-travel. Context is the right tool. If state grew, I'd consider Zustand / Redux Toolkit / TanStack Query.
17. Migrate Products.tsx fetch logic to TanStack Query.
useQuery(['products', q], () => api('/api/products?q=' + q)) for reads, useMutation for adjust/delete/create with queryClient.invalidateQueries(['products']) on success. Removes manual reload() calls and adds caching/background refetch.
18. Why both RequireAuth and PublicOnly?
RequireAuth keeps unauthed users out of app pages. PublicOnly keeps authed users out of login/signup so the form doesn't flash before redirect.
19. The Vite proxy — why?
SPA on :5173, API on :4000. Without the proxy, dev would need CORS. The proxy makes the SPA think the API is same-origin.

Security

20. Top 3 security risks here?
(1) No CSRF token on state-changing routes (sameSite mitigates but doesn't eliminate). (2) No rate limiting → credential stuffing / signup spam. (3) Float for money → arithmetic bugs that look like fraud.
21. Add Google SSO. Where do you hook in?
auth.routes.ts: add /auth/google/start (redirect to Google) and /auth/google/callback (exchange code, find-or-create user scoped to existing email or new org, signSession + setSessionCookie). Add provider and providerId columns on User. Decide the "first signup with SSO" → org-creation rule.
22. CSRF — explain the mitigation and where it's incomplete.
sameSite=lax means the cookie is only sent on top-level same-site navigations / first-party requests. Cross-site POSTs don't include it. But same-origin malicious scripts could still issue requests. Defense in depth: CSRF token (double-submit cookie or per-form) on mutations.

Scaling & Ops

23. Move from SQLite to Postgres — what changes?
provider = "postgresql", DATABASE_URL, regenerate. Review migrations (SQLite is lax about ALTER). Add connection pooling (PgBouncer / Prisma Accelerate). Push some computations to SQL aggregates.
24. How do you horizontally scale this?
Stateless API (JWT in cookie, no in-memory session) → N replicas behind a load balancer. DB is the bottleneck → vertical scale + read replicas for dashboard. Redis for rate limits and any caching.
25. CPU spikes during dashboard loads — investigate?
Add a logger + request IDs. APM via OpenTelemetry. Slow query logs from Postgres. The dashboard route iterates all products in JS — push to SQL count / sum aggregates.
26. Add an audit log without breaking the API contract.
New AuditEvent model (id, organizationId, userId, action, entityType, entityId, metadata, createdAt). Wrap mutation handlers (or use Prisma middleware) to write events. Move to an async queue (BullMQ) if it becomes a hot path.
27. Support user invites by email. Design it.
Invite { id, email, organizationId, role, tokenHash, expiresAt, acceptedAt? }. POST /invites (ADMIN) creates row + emails https://app/accept?token=…. Accept route validates token, creates User with that role + org, signs in. Token is single-use, hashed at rest, 7-day expiry.

Architecture / "Principal" Questions

28. What would you redesign if you started over?
  • Postgres from day 1.
  • Wrap Prisma with an org-aware extension so tenancy can't be forgotten.
  • Money in integer cents (or Decimal).
  • TanStack Query + React Hook Form.
  • Replace bespoke fetch wrapper with ky or axios.
  • Add an audit log and a central event emitter.
  • Adopt OpenAPI (or tRPC) so client/server schemas are generated, not duplicated.
29. Where's the dead code / overkill?
/api/auth/me exists but the SPA only uses /session. Two endpoints doing nearly the same thing — consolidate or document the difference.
30. What single test would you write first?
An end-to-end test: sign up org A, sign up org B, A creates product X, B tries GET /products/X → expect 404. That one test guards the most catastrophic failure mode (cross-tenant leak).
